理念

申明式编程和命令式编程的区别：他们最大的区别是命令式编程除了要相应事件还得处理消息的状态，例如TextField我们得实现代理来监听文字输入的变化，但是声明式编程我们不需要编写代理去处理消息，每一次文字的输入都是信号，我们通过过滤信号从而得到我们想要的消息并作出相应的响应。

函数式反应型编程函数式反应型编程是两个声明式编程的子范例(函数式+反应式)的组合。
反应式编程：在表处理方面十分强悍。假设我们有一个表格A:她是用来纪录其他两个表格(表格B、表格C)的和。当表格B或C当中任意一个值发现变化时，这些变化都会通过表实时改变表格A的值。总之，我们定义好了A是B和C的和，不管发生了什么,A会一直响应B或C的变化,永远都是B与C的和。

函数式编程：函数式编程中，对于同样的输入，一个函数f始终会给出同样的输出，不存在’可变的状态’。这听起来有点不可思议，我们可都是依靠状态的多变性来编写程序啊。在这个给变量赋值之后就不可以重新赋值的世界里，想想都觉得不可思议。函数式编程在很多方面显得不太实用。很多编程涉及到用户的输入、网络输入/输出等等，都不太容易使用函数范式来构建。这也是为什么函数式编程作为函数式反应型编程的一部分而出现的原因。因为函数式反应型编程是命令行编程与函数式编程两者相互妥协的最佳平衡点。她让我们有鱼与熊掌兼得的意思。

维基百科：

声明式编程（英语：Declarative programming）是一种编程范型，与命令式编程相对立。它描述目标的性质，让电脑明白目标，而非流程。声明式编程不用告诉电脑问题领域，从而避免随之而来的副作用。而指令式编程则需要用算法来明确的指出每一步该怎么做。

流和序列

流是值的序列化的抽象，你可以认为一个流就像一条水管，而值就是流淌在水管中的水，值从管道的一端流入从另一端流出。当值从管道的另一端流出的时候，我们可以读取过去所有的值，甚至是刚刚进入管道的值(即当前值)。

前面高阶映射做了以下步骤

  • 将数组转换成为一个序列类型的流
  • 对流进行映射得到一个新的流
  • 将新的流转为数组

而高阶折叠中就是将序列进行了链试调用，ReactiveCocoa具有左折叠和右折叠的概念，做折叠就是从头到尾遍历数组，反之称为右折叠。

信号

信号是另一种类型的流。与序列流相反，信号是push-driven的。新的值能够通过管道发布但不能像pull-driven一样在管道中获取，他们所抽象出来的数据会在未来的某个时间传送过来。
Pull-driven: 在创建信号的时候，信号不会被立即赋值，之后才会被赋值(举个栗子：网络请求回来的结果或者是任意的用户输入的结果)
Pull-driven：在创建信号的同时序列中的值就会被确定下来，我们可以从流中一个个地查询值。

信号的三种状态：
Next Values：下一个发布到管道内的值。
Error Value: signal无法完成。
Completion Values：signal 成功完成。

指令

指令，RACCommand类的代表，创建并订阅动作的信号响应，可以很容易地实现一些用户与应用交互的边界效果。
指令（行为触发的的）通常是UI驱动的，比如按键的点击。指令就可以通过信号自动禁用，这种禁用状态呈现在UI上就是禁止该指令相关联的任何操作。

[RACSignal empty]这里我们需要返回一个信号让属于RACCommand的executionSignal管道(pipe)下发出去。这个信号代表button按下时一些任务需要被处理。在这个处理信号没有返回一个’complete value’(‘empty ‘会立即返回一个’complete value’)之前button将会保持不可用状态.因为这个例子中我们只是打印了一下，所以这里我们只返回一个empty信号。
